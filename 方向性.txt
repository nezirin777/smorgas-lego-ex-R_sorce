# Chaika スキンスクリプト リファクタリング方針

## 目的

本プロジェクトでは新機能追加は行わず、既存機能の整理と改善を目的とする。

主な目標は以下の通り。

- 可読性向上
- 保守性向上
- 軽量化
- パフォーマンス改善
- 現代Firefox環境への最適化
- script.js の肥大化解消
- 将来的な修正・拡張の容易化

既存ユーザーから見た動作やUIは可能な限り維持する。

---

# 対象環境

## サポート対象

- Firefox 最新版
- Floorp
- Zen Browser
- Waterfox

## サポート対象外

- Firefox 52
- 旧Firefox ESR
- Pale Moon
- ES5環境

これら旧環境向けの互換コードは削除候補とする。

# 現状認識

本コードは完全なレガシーコードではなく、

- Functionベース
- Namespaceベース
- Classベース

が混在している。

例

function AjaxGet() {}

var Nodes = {};

class StyleSheet {}

そのため、

「全コードを一括Class化する」

のではなく、

「整理しながら段階的にClass化する」

方針を採用する。

# 変数宣言ポリシー

## 基本方針

本リファクタリングでは、原則として var の使用を減らし、
ローカルスコープでは const / let を使用する。

ただし、トップレベル（グローバル）に公開される一部のマッピング変数については、
互換性維持のため var を継続使用する。

---

## ローカル変数

対象

- 関数内部
- メソッド内部
- ブロックスコープ内
- ループ変数

方針

- 再代入しない → const
- 再代入する → let

例

const node = document.querySelector("#main");

let count = 0;
count++;

---

## グローバルマッピング変数

対象例

- Analyse
- SoundUnit
- Clipboard
- Punycode
- Nodes
- ResNodes

方針

互換性維持のため var を維持する。

例

var SoundUnit = {};
var Clipboard = {};

---

## var を維持する理由

### 1. window への自動アタッチ

トップレベル変数は宣言方法によって挙動が異なる。

var

var SoundUnit = {};

↓

window.SoundUnit

として参照可能。

const / let

const SoundUnit = {};

↓

グローバルスコープからは参照できるが、

window.SoundUnit

は undefined となる。

Chaika本体や関連スクリプトが

window.SoundUnit
window.Clipboard

などの形式で参照している可能性があるため、
機械的な const 化は行わない。

---

### 2. 再ロード時の安全性

const は同一コンテキストで再宣言されると即座に例外となる。

const SoundUnit = {};
const SoundUnit = {};

↓

SyntaxError:
Identifier 'SoundUnit' has already been declared

一方 var は再宣言が許容される。

var SoundUnit = {};
var SoundUnit = {};

↓

エラーにならない。

Chaika のようなアドオン環境では、

- スクリプト再読み込み
- 多重ロード
- 旧コードとの共存

が発生する可能性があるため、
トップレベルでは var の方が安全である。

---

### 3. 過去の修正履歴との整合性

CHANGELOG.md には以下の履歴が残っている。

「トップレベルのconstをvarに書き換えました」

これは過去に実際に発生した

- ロードエラー
- 名前衝突
- 実行コンテキスト問題

への対処として行われた修正であり、
リファクタリング時も尊重する。

---

## 適用ルール

○ 変換する

var node = ...
↓
const node = ...

var i = ...
↓
let i = ...

○ 維持する

var Analyse = {};
var SoundUnit = {};
var Clipboard = {};
var Punycode = {};
var Nodes = {};
var ResNodes = {};

---

## 結論

本プロジェクトでは

- ローカル変数 → const / let
- グローバル公開マッピング → var

を正式方針とする。

単純な「var撲滅」は行わず、
実行環境との互換性を優先する。

# Class化方針

## 状態を持つオブジェクト

現状

var Bookmark = {
    _data: {},
    save(){},
    load(){}
};

↓

class Bookmark {

    constructor() {
        this._data = {};
    }

    save() {}

    load() {}
}

const bookmark = new Bookmark();

---

## 状態を持たないユーティリティ

現状

var UrlUtil = {
    normalize(){},
    encode(){}
};

↓

class UrlUtil {

    static normalize() {}

    static encode() {}

}

---

# DOM管理方針

現在は以下の独自管理層が存在する。

- _doc
- Nodes
- ResNodes

これらは依存範囲が広いため、初期段階では触らない。

まず利用側の整理を優先し、

Nodes / ResNodes の見直しは後半フェーズで実施する。

# リファクタリング工程

## Phase 1：現状把握

対象コードについて以下を整理する。

- 役割（責務）
- 呼び出し元
- 依存関係

---

## Phase 2：安全な整理

実施内容

- const / let化
- 未使用変数削除
- 未使用関数削除
- 重複処理削除
- 旧互換コード削除
- 不要なDOM取得削除

目的

- コード量削減
- 可読性向上

---

## Phase 3：小規模Class化

優先対象

1. AjaxGet
2. StyleSheet周辺
3. 小規模ユーティリティ

ここでClass化のルールを確立する。

---

## Phase 4：Popup系整理

対象

- Popup
- ResPopup
- IDPopup
- NamePopup
- TrackbackPopup
- UrlPopup
- ImagePopup
- VideoPopup

目標

class PopupBase {}

を導入し、共通処理を集約する。

---

## Phase 5：UI系整理

対象

- Bookmark
- AutoReload
- Digest
- AutoScroll
- FindBox
- PageScroller

主に責務整理とClass化を行う。

---

## Phase 6：Thread系整理

対象

- ThreadDocument
- Name
- ID
- Trackback
- ResNumber
- ResImage

依存関係が多いため後回しとする。

---

## Phase 7：Nodes / ResNodes整理

対象

- Nodes
- ResNodes

全体への影響が非常に大きいため慎重に進める。

必要に応じて

var Nodes

↓

class NodesManager

への移行も検討する。

---

## Phase 8：ファイル分割

最終目標は責務ごとの分離。

core/
├ AjaxGet.js
├ StyleSheet.js

popup/
├ PopupBase.js

├ ImagePopup.js

ui/
├ Bookmark.js
├ AutoReload.js

thread/
├ ThreadDocument.js

util/
├ Clipboard.js
├ ReplaceStr.js
